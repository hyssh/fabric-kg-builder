Background

Why a knowledge graph — and why this is hard

Enterprises sit on thousands of manuals, guides, and reports. Vector search alone returns passages, not connected facts. A knowledge graph captures the relationships — which part belongs to which device, which symptom a cause produces, which step a procedure requires — so an AI agent can reason over them. The hard part isn't drawing boxes; it's building a graph that's domain-fit and densely connected enough to actually answer questions.

📚

The input problem

Real documents mix prose, tables, figures, and part numbers. Extraction must handle all of them — Document Intelligence pulls tables as HTML and figures as images to Blob storage.

🧮

The sparsity problem

LLMs extract per-section, leaving the graph fragmented — device models disconnected from their own parts; symptoms with no causes. A graph with islands answers nothing.

🎯

The fit problem

A generic, one-size-fits-all schema retrieves poorly. The graph model should be shaped by your industry, domain, and the questions you actually need answered.

Capabilities

What it does

One CLI takes you from raw files to a live, queryable graph in Microsoft Fabric.

🤖

LLM enrichment

Extracts entities, relationships, and evidence from documents via Azure OpenAI / Microsoft Foundry, guided by your domain brief.

🔗

Graph densification

A deterministic pass links each document's device model to its parts & procedures and connects Cause→Symptom→Resolution — no extra LLM cost.

🌐

Multi-type ontology

Deploys one Fabric entity type per real domain type with typed relationships, so the Ontology Explorer shows a rich graph, not one generic box.

📊

Lakehouse + AI Search

Structured graph and ontology land in a schema-enabled Lakehouse; document text and embeddings power hybrid retrieval in Azure AI Search.

🧠

Data Agent grounding

Auto-generates a Fabric Data Agent instruction file from the live graph and your sample questions, so natural-language → GQL actually returns rows.

⚙️

fabric-cicd deployment

Deterministic, repeatable deploys to Lakehouse, Ontology, and Search across dev / test / prod environments.

Install & prerequisites

Requires Python 3.10+, an Azure subscription with a Microsoft Fabric workspace (schema-enabled Lakehouse + Ontology), Azure OpenAI / Foundry, Azure AI Search, Document Intelligence, and Blob storage. Auth uses az login (DefaultAzureCredential).

install
# clone and install (editable, with dev extras)
git clone https://github.com/hyssh/fabric-kg-builder
cd fabric-kg-builder
pip install -e .[dev]

# authenticate & configure secrets
az login
cp .env.example .env   # fill in endpoints

fabric-kg --help
  • Self-documenting CLI — every command has examples in --help
  • Secrets stay in a local .env (never committed)
  • Per-environment config under ontology/environments/
  • Deploys via fabric-cicd to your Fabric workspace
  • 918+ unit tests; structured validation gates on every build

Use it with the GitHub Copilot CLI

Every fabric-kg command ships rich --help — a pipeline overview, parameters with defaults, worked examples, and a contact line. That makes it ideal to drive conversationally from the GitHub Copilot CLI: Copilot reads the help, understands the flow, and runs the right command for you.

in the GitHub Copilot CLI

> run fabric-kg --help and summarize the pipeline

# then ask it to do the work
> set the domain for field-service troubleshooting,
  industry manufacturing, using my questions file
> enrich the PDFs in sample_data, then densify
> deploy a multitype ontology to dev
what Copilot runs for you
fabric-kg set-domain --industry manufacturing \
  --business-domain field-service \
  --questions-file sample_data\surface_questions.txt \
  --prompt "Surface hardware troubleshooting..."

fabric-kg enrich --input sample_data\Surface_Troubleshootings
fabric-kg densify --input build\enriched --out build\enriched_dense

Best practices

Design once, iterate cheaply

The biggest driver of graph quality is the up-front domain brief. Get these right and you minimize the expensive cycle of re-design and re-deployment.

Start from a domain template

Name the entity types and relationships that fit your industry. The Surface corpus uses a field-service template: Device, DeviceModel, Component, Part, PartNumber, Procedure, Step, Tool, Symptom, Cause, Resolution.

Healthcare, legal, and finance starters are in the README's Domain Template Playbook.

Declare industry & business domain

set-domain requires --industry and --business-domain. These shape which types the model proposes and how the graph is organized.

They flow into enrichment and into the auto-generated Data Agent grounding.

Lead with sample questions

Pass --questions-file with 3–5 questions the graph must answer. These are the single biggest lever on ontology quality — they tell the model exactly which entity types and relationships are needed so every question is answerable.

They're also recycled as Data Agent few-shots automatically.

Always densify before compiling

Run fabric-kg densify between enrich and compile-data. It connects the islands extraction leaves behind — without another LLM pass.

On the Surface corpus this took relationships from 3,715 → 32,118 and isolated symptoms from 327 → 8.

Iterate without re-enriching. Enrichment is the slow, costly stage — run it once. Densify, compile, and deploy-ontology --multitype all reuse the enriched data, so refining your model and re-deploying takes minutes, not hours. Test → inspect → tune densify or grounding → re-deploy. No re-enrichment required.

FAQ

Common questions

Why does the Fabric Ontology Explorer show one box per type, not per record?

The Explorer is a schema view: it draws one box per entity type, with all instances bound behind it from Lakehouse tables. With --multitype you get one box per real domain type (Component, Procedure, Symptom…) instead of a single generic node.

Do I have to re-run enrichment every time I change the model?

No. Enrichment is run once. Densify, compile, and deploy all reuse the enriched JSON, so iterating on the graph model and re-deploying takes minutes.

My Data Agent returns “no data found” even though the graph has data. Why?

Usually a name mismatch (exact-match on “Surface Pro 10” vs the real “Surface Pro 10 for Business”) or a sparse graph. Run densify, and paste the auto-generated data-agent-instructions.md into your agent — it enforces CONTAINS matching and short, single-hop queries.

Can I use it for non-hardware domains?

Yes. The Surface template is just the reference. Define your own entity types and relationships for healthcare, legal, finance, or any document-heavy domain — the README's Domain Template Playbook has starting points.

What deploys the Fabric items?

fabric-cicd. Deployments are deterministic and repeatable across dev / test / prod, reading per-environment config from ontology/environments/.
