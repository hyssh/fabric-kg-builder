### 2026-06-24T17:01:06-07:00 — Sprint 1 Ontology Compiler Implementation

- **The compiler must own both disk-write and REST-payload modes from a single `_iter_parts()` internal method.** Computing parts once as `(relative_path, content_dict)` pairs and then either writing them to disk or serializing them to Base64 REST payloads avoids duplication and ensures the two outputs are always in sync. Never maintain two separate codepaths for the same logical content.
- **`blob_url` type in model.yaml → `"type": "String", "format": "uri"` in Fabric definition JSON.** The type name `blob_url` is the model.yaml signal; the compiler must check `ptype == "blob_url"` explicitly (not by property name) and inject `"format": "uri"`. A plain `type: string` with a name of `blob_url` would silently miss the format annotation.
- **UUIDv5 derivation: namespace first, then type+table key.** The GUID for a data binding or contextualization is `uuid5(uuid5(NAMESPACE_DNS, "FabricKG"), f"{type_name}:{table}")`. This two-level derivation means the ontology name acts as a salt, so a differently-named ontology would produce different GUIDs. Encode this as a module-level constant `_ONTOLOGY_NS` so every GUID derives from the same root.
- **`inverseTypeId` must be looked up from `inverseName` in ids.lock, not hard-coded.** The forward relationship (e.g., `has_component`) carries `inverseName: component_of`; the compiler looks up `rel_ids["component_of"]` to get the numeric ID. If the inverse type has no lock entry, the compiler silently omits `inverseTypeId` rather than raising. Validation guarantees it has an entry (since it's also declared in model.yaml).
- **Validation must run at construction, not at compile time.** Fail fast at `OntologyCompiler.__init__` so callers catch errors before any I/O occurs. All four checks (entity ID present, relationship ID present, no duplicate IDs, source/target types exist) run in one pass in `_validate()`.
- **Module-scoped pytest fixtures avoid re-compiling 38-type ontology on every test.** `compiler`, `rest_parts`, and `compiled_dir` are all `scope="module"`. Function-scoped `tmp_path` is fine for validation-error tests since those construct tiny inline models. Use `tmp_path_factory.mktemp()` (session-scoped) for the `compiled_dir` fixture that uses a module-scoped `compiler`.
- **`definition.json` (top-level manifest) must NOT include itself in the parts list.** Only `.platform`, EntityTypes, and RelationshipTypes files appear in `parts`. The manifest lists everything it points to; adding itself would be circular and confuse the Fabric REST API consumer.
- **Property type map needs an explicit `blob_url` entry.** Without it, `_PROP_TYPE_MAP.get("blob_url", "String")` would fall through to the default "String" but skip the `format` injection, since that check is inside an `if ptype == "blob_url"` branch. The map entry is for clarity; the `format` logic is separate.
- **`typeFilterColumn`/`typeFilterValue` should both be included or both omitted.** If a binding has `typeFilterColumn: asset_type` but `typeFilterValue: ""` (ImageAsset — returns all rows), include both fields in the JSON output. Fabric's schema may reject a filter column with no corresponding value field.
- **Coverage for the compiler module hit 100% on the first run.** All branches exercised by the 44 new tests. This validates that the test suite covers the validation, property building, entity, relationship, platform, and REST serialization paths completely.

### 2026-06-24T16:09:30-07:00 — Sprint 1 model.yaml + ids.lock.json Implementation

The actual file must enumerate every entity type and relationship type from §2.2/§2.3, including all document-evidence, visual-evidence, and retrieval types. Build the full list from §2 tables, not from the snippet.
- **Materialized inverse relationships must be declared as explicit relationship type entries.** `component_of`, `part_of`, `step_of`, `shows`, `identified_by` need their own entries in both `model.yaml` (as relationship types with a reversed data binding) and `ids.lock.json` (in the 200x range). They cannot just be properties or compiler output — the compiler needs them declared to emit `inverseTypeId` in relationship definition.json files.
- **Visual type blob_url gate is a hard compiler error.** Every visual entity type (ImageAsset, Figure, VisualRegion, Image, Screenshot, Diagram, Photo) MUST carry `type: blob_url` (not `type: string`). The YAML `type` value is exactly `blob_url` — the compiler uses this literal string to emit Fabric property format `uri`.
- **search_aliases lives on entities table, not as a list<string> in YAML.** The SPEC-003 §3.3 schema uses `type: string` for all property types in model.yaml (including multi-value fields like search_aliases). The `list<string>` type from SPEC-002 is a Parquet/pyarrow type, not a model.yaml property type. Keep property types in model.yaml as string even for list-valued columns; document the column type in the description.
- **BRG-003 gate requires entity_id + canonical_key + search_aliases on all support-domain entities.** All 11 support-domain entity types (Device, Model, Component, Part, PartNumber, Tool, Symptom, Cause, Resolution, Procedure, Step) need these three properties explicitly declared in model.yaml. The compiler gate checks by name.
- **ids.lock.json ranges: document-evidence starts at 100, retrieval at 200, inverses at 2000...200.** Confirmed by §4.2. Support-domain entity types 001–011, document/visual types 100–119, retrieval types 200–203. Relationship type ranges mirror this: support-domain 001–009, evidence/visual 100–121, inverses 200–204. Keep contiguous and comment the range boundaries in the JSON.
- **Verification script should test blob_url property type, ID ranges, duplicate IDs, and bridge relationships in one pass.** Combining all checks into one script (as done here) reveals whether the file is deploy-ready in a single run. Add this as the standard ontology sanity check pattern.
- **provenance module has no entity types — just relationship types.** The module entry still needs an `entityTypeNames: []` field (not omitted) to satisfy the schema parser. An empty list is valid YAML; omitting the key may cause KeyError downstream.
- **Document and DocumentChunk bind to different tables.** `Document` → `source_files` (entityIdColumn: `source_file_id`); `DocumentChunk` → `chunks` (entityIdColumn: `chunk_id`). These are easy to confuse. Always verify the entityIdColumn matches the Parquet table's primary key per SPEC-002 §4.1.

### 2026-06-24T15:41:07.842-07:00 — fabric-cicd Required Primary + kg_lakehouse Rename

- **`fabric-cicd` is REQUIRED, not optional.** The previous "wrap both, fall back to REST" framing was wrong — it implied fabric-cicd was interchangeable with direct REST calls. fabric-cicd is the *mandatory* primary publication path. The Fabric REST API is a fallback only for item-level granularity that fabric-cicd cannot satisfy. Document it that way, not as "either/or."
- **Prerequisite installation must be explicit in spec.** `pip install fabric-cicd` must be stated as a CLI prerequisite in the spec and cross-referenced to `docs/REQUIREMENTS-001-cli-prerequisites.md`. The spec previously assumed fabric-cicd was available without instructing how to get it.
- **The deploy flow is compile → dist/ → fabric-cicd publish.** The CLI packages artifacts to `dist/` first (compile-data, compile-ontology, compile-search), then invokes fabric-cicd to push `dist/` to the Fabric workspace. This two-step pattern (package, then publish) must be explicit in each deploy-* internal steps list so implementers understand the correct sequence.
- **`deploy-search` is in-MVP, not optional.** It was marked *(optional)* and gated by an `if: ENABLE_AI_SEARCH == 'true'` CI/CD condition. That framing was wrong per the PRD and coordinator decision. Remove the gate; it deploys as part of the standard MVP pipeline.
- **Lakehouse name is `kg_lakehouse`, not `fabrickg_lakehouse`.** The rename has already happened in the workspace. The spec and all downstream config must use `kg_lakehouse` (item ID `44444444-4444-4444-4444-444444444444`, workspace `11111111-1111-1111-1111-111111111111`). Store IDs in `ontology/environments/dev.json`; never hardcode them in source code.
- **`FabricDeployer` wrapper defaults to fabric-cicd.** The wrapper class still exists as an abstraction layer, but its default code path must be fabric-cicd, not REST. REST is a conditional fallback branch, not the happy path.



- **`chunks` has no `canonical_key`, `search_index_name`, or `aliases` columns.** These were the three phantom columns the previous §12 draft claimed as chunk properties. They do not exist in SPEC-002. Always verify column names against Fenster's Appendix A schemas before writing a data binding.
- **The real source columns are `chunks.related_entity_ids` and `chunks.entity_search_keys`.** `related_entity_ids` carries stable opaque entity IDs (feeds `entity_ids` filter); `entity_search_keys` carries the denormalized, human-readable alias list (feeds `entity_aliases` keyword search). These are the only two chunk columns that touch entity identity.
- **AI Search index fields are DERIVED, not raw.** `entity_ids`, `canonical_key`, `entity_aliases`, and `graph_path` appear in the AI Search document, but they are computed at index-build time via compile-data JOINs. The ontology spec must state "derived at index-build time" rather than implying those fields come directly from chunk columns.
- **`entities.canonical_key` vs `chunks.canonical_key` — always qualify with the table name.** The property `canonical_key` exists on the `entities` table (as a direct column) and on `SearchIndexRecord` ontology nodes (as a derived field JOINed from `entities.canonical_key`). It does not exist on `chunks`. When writing data bindings, always prefix the table: `entities.canonical_key`, not just `canonical_key`.
- **`entities.search_aliases` is the source; `chunks.entity_search_keys` is the materialized copy.** The normalization pipeline writes `entities.search_aliases` = `[canonical_key] + [display_name.lower()] + [alias.lower() for alias in aliases]` at entity compile time, then denormalizes it onto `chunks.entity_search_keys` at compile-data time. Feeding AI Search from `entity_search_keys` is correct; feeding it directly from `entities.search_aliases` at query time bypasses the materialization and creates a runtime dependency.
- **`search_index_name` is deployment config, not a SPEC-002 column.** Which AI Search index a chunk lives in is determined at deploy-search time from the `fabric-kg.yaml` config, not from a Parquet column. It should never appear in a `dataBinding.additionalColumns` entry. The BRG-002 gate that previously required it was wrong and has been corrected.
- **BRG validation gates must name SPEC-002 column names exactly.** BRG-001, BRG-002, and BRG-003 now reference the correct canonical column names. Any future BRG gate that names a node property must be verified against SPEC-002 Appendix A before being added.
- **Cross-spec column references need a "SPEC-002 source column" column in tables.** The §12.3 table format (node type, property, type, required, SPEC-002 source column, purpose) is the right pattern. The "SPEC-002 source column" column forces every row to name exactly where the data comes from — it's what caught the phantom column bug.

### 2026-06-24T18:04:21-07:00 — Sprint 2 Bridge Validation (compile-ontology-e2e)

- **`bridge_validation.py` is a pure function module: model dict in, list of violations out.** No I/O, no side effects. `validate_bridge(model)` accepts the already-unwrapped model dict and returns `list[BridgeViolation]`. The CLI layer handles formatting and exit codes. This separation makes it trivially unit-testable and reusable by future `fabric-kg validate` commands.
- **`_bound_columns(entity_type)` collects ALL column names in a data binding, including `entityIdColumn`, `displayNameColumn`, and `additionalColumns[].column`.** This is intentional — any binding mechanism that exposes the column to the Fabric Ontology runtime counts. When testing for "missing" column, remove ALL references (entityIdColumn, displayNameColumn, and additionalColumns) or the check will still pass. Learned this from BRG-002/BRG-006 test failures.
- **BRG-009 is a Warning gate, not an Error gate.** Support-domain entities (Device, Model, Component, etc.) that have no outbound `evidenced_by` or `shown_in` relationship type are not errors — the shipped model intentionally only declares one instance of each bridge relationship. The 9 BRG-009 warnings are expected diagnostic output, not blockers.
- **Bridge gate wiring in compile-ontology: run AFTER `compiler.compile()`, before the summary.** If bridge errors are found, exit 5 (same code as model validation errors) after printing the error list. This ensures the Fabric definition tree is still written to disk before the exit — callers can inspect the output even when bridge validation blocks. Only errors block; warnings don't.
- **BRG-005 checks that `inversePolicy` key IS PRESENT in the relationship dict (any value including "none" is valid).** The gate is not checking for a specific policy — it's ensuring the field was not accidentally omitted from model.yaml. All three bridge relationships (`evidenced_by`, `shown_in`, `indexed_as`) pass in the shipped model because they all declare `inversePolicy`.
- **Deliberately broken test models need dedicated IDs lock files.** The CLI integration test for "broken binding → exit 5" must write both a `broken_model.yaml` and a `broken_ids.json` with matching entity/relationship type entries. The ids.lock validation runs BEFORE bridge validation; a missing ID causes exit 5 with a different error message than a bridge gate failure. Use subset IDs from the real lock file.
- **`_make_minimal_model()` helper is the right pattern for targeted gate breakage tests.** One baseline correct model function + per-test mutations = isolated, readable tests. The helper returns a plain Python dict (not a YAML string), so mutations are clean Python list comprehensions without string parsing. Fixture scope is function (not module) because each test mutates the model.
- **BRG-004 checks BOTH the property declaration AND the data binding column.** A visual entity type needs to declare `blob_url` property with `type: blob_url` AND have it bound to a column in `additionalColumns`. Missing either is an error. The two-part check catches cases where the property exists in YAML but the compiler cannot populate it from the Lakehouse table.
- **The compile-ontology summary now includes `Bridge validation: N errors, M warning(s)`.** This is the human-readable confirmation line the user asked for. The format is consistent with the existing summary block style and easy to grep in CI logs.



- **`sys.exit(N)` works correctly inside Click CliRunner tests.** Click wraps `SystemExit` exceptions in the result object and surfaces them as `result.exit_code`. Using `sys.exit(5)` for validation errors (not `raise click.Abort()`) is the right pattern when the exit code must be testable and distinguishable from generic failures.
- **Windows console (cp1252) rejects Unicode box-drawing characters at runtime.** `─` (U+2500) causes `UnicodeEncodeError` on Windows even with `click.echo`. Use plain ASCII `"-" * 60` for separator lines in CLI output. Keep Unicode box chars only in documentation or test output that explicitly sets UTF-8.
- **`is_flag=True, default=True` does NOT register `--flag-name` as a recognized option in Click.** Click treats the explicit name as the "toggle to True" path, but since it's already the default, Click omits it from the options list. Use the pair form `--mock/--no-mock` with `default=True` instead — this registers both forms and documents the override cleanly.
- **Stale `.pyc` cache causes `NameError` at collection time when a source file has been regenerated between sessions.** On Windows, deleting all `__pycache__` directories (`Get-ChildItem -Recurse -Filter "__pycache__" | Remove-Item -Recurse`) unblocks pytest collection. Never diagnose a `NameError` as a code bug before first clearing caches.
- **`load_fabric_ids()` helper should bypass the full `load_config` flow.** The full config loader requires `AZURE_AI_FOUNDRY_ENDPOINT` to be set. compile-ontology and deploy-ontology only need `workspace_id` and `lakehouse_item_id`. A dedicated lightweight helper reads just the env JSON without triggering the foundry endpoint assertion — keeps offline/CI workflows unblocked.
- **`FabricDeployer.from_build_dir` + `deploy(mock=True)` is the correct seam for fabric-cicd.** The class already has a clear `# TODO (fabric-cicd primary path)` comment in `deploy()`. The mock path is fully tested offline. When fabric-cicd is wired, only the `deploy()` method body changes — no interface change, no test changes.
- **`definition.json` parts count is one more than `_iter_parts()` length.** `_iter_parts()` returns N parts; `compile()` also writes `definition.json` itself. The parts count in the CLI summary is `len(parts) + 1` to account for the manifest. Report this to users so the count matches what they see on disk.
- **The deploy-ontology `--dist` default should match compile-ontology's `--out` default.** Both default to `build/ontology`. Mismatching defaults (dist was `dist`) forces users to remember to pass `--dist build/ontology` on every deploy call. Keep defaults consistent across the pipeline stages.



- **SPEC-001 (snake_case) and SPEC-003 (camelCase) both define the per-env JSON format with slightly different field sets.** Combined them into one richer snake_case schema that satisfies both specs. Code that reads these files should use the SPEC-001 keys; the SPEC-003 camelCase aliases are for reference only.
- **Lakehouse item ID is legitimately unknown at config time.** INFRA-001 flags this as an open action item. The right pattern is a clearly-marked `"<dev-lakehouse-item-id>"` placeholder with a `_comment` directing the team to fill it in after workspace provisioning. Don't block the file on one unknown.
- **Foundry endpoint format is not fully verified.** The standard pattern `https://<account>.services.ai.azure.com/api/projects/<project>` is the documented format but the exact URL must be confirmed in the Azure AI Foundry portal. `_comment_endpoint` fields on Foundry and Cognitive Services sections document this explicitly so the first engineer to use them knows to verify.
- **`embedding_dimensions: 1536` belongs in the env JSON even though it is technically stable across envs.** It is the coupling constant between the embedding model deployment and the AI Search vector field. Putting it here alongside the deployment name makes the coupling explicit and discoverable at the env layer, even if it never actually changes.

### 2026-06-24T12:42:17.255-07:00 — SPEC-003 §12 Research Fold (RESEARCH-001 → Graph-to-Search Bridge)

- **GraphRAG Local Search / OmniRAG are the named reference patterns.** Always cite them by name in the spec. Engineers unfamiliar with the architecture need to know why the two-phase flow looks the way it does, and naming the reference implementations gives them a starting point.
- **Bounded traversal is not optional.** Uncapped graph traversal silently blows AI Search filter size limits. The 2-hop / 10–20 entity cap must be in the spec as a hard rule, not a recommendation. BRG-004's hop count and entity-cap rule should be enforced by the validator, not left to the agent author.
- **entity_id and aliases belong on ALL support-domain entity types, not just retrieval nodes.** The phase 2 AI Search query is built from the domain entity's `entity_id` (for `search.in()` filter) and `aliases` (for the BM25 keyword query string). If these are missing from a domain entity, the bridge silently degrades to a pure vector query — no filter, no keyword boost.
- **graph_path is a first-class output.** Serializing the traversal path at query time (not reconstructing it later) is essential for LLM citation. The format `EntityType "label" --[rel]--> NodeType "label"` is simple enough for the LLM to quote in citations and precise enough for engineering debugging.
- **OneLake indexer Parquet limitation is a gotcha that always bites.** RESEARCH-001 §5 is explicit: the OneLake indexer reads only the Files location, not Parquet/Delta. This must be in the spec so every engineer touching `deploy-search` knows to use Push API or a Fabric Data Pipeline, not a naive OneLake indexer config.
- **BRG validation gates must check that bridge relationships resolve to existing records.** It is not enough to check that the relationship type exists; BRG-006/007/008 check that the target nodes actually exist and have the required keys populated. This prevents silent dangling edges in the ontology.
- **Align with SPEC-002 by name, not by copy.** Repeating Fenster's column definitions in §12 creates drift. The right pattern is a named reference ("SPEC-002 normalization rules apply") plus a `dataBinding.additionalColumns` entry pointing to the Parquet column.

### 2026-06-24 — SPEC-003 Ontology and Deployment Spec

- **One ontology, five modules.** The split into support-domain, document-evidence, visual-evidence, retrieval, and provenance is clean and traversable. Keep modules as organizational groupings only — they share the same Fabric ontology item and the same `ids.lock.json`.
- **`blob_url` is a property, not an edge.** `stored_at` from the PRD §9 traversal example resolves as a `blob_url` property on the ImageAsset/Figure/VisualRegion node rather than a separate relationship edge. This keeps visual asset resolution simple: one property read, no join.
- **Inverse policy must be explicit.** The lesson from the AI Search prototype is that direction matters. Every relationship type in `model.yaml` must declare `inversePolicy`. Default should be `none`, not implicit materialization.
- **Deterministic GUIDs for bindings.** Data binding GUIDs and contextualization GUIDs should be derived with UUIDv5 from the ontology name + entity type name + table name. This ensures the same GUID appears in every environment without storing it separately.
- **Placeholder parts are always emitted.** The compiler should emit a definition.json for every type in model.yaml regardless of whether Parquet data exists. Empty bindings with `rowCount: 0` are valid Fabric definition parts and allow model-first deployment.
- **ID ranges work.** Entity type IDs starting with `1`, relationship type IDs starting with `2`. Simple, memorable, and easy to validate with a single prefix check. Inverse relationship IDs go in the `200` block of the relationship range.
- **Environment config injects at compile time, not at deploy time.** Lakehouse ID and workspace ID should be baked into data binding JSON files during `compile-ontology --env {env}`, not patched during `deploy-ontology`. This produces self-contained artifacts per environment.
